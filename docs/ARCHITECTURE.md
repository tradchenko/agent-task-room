# Архитектура agent-task-room

## Цель

`agent-task-room` создаёт общее пространство решения задачи для двух людей и двух агентов, даже если:

- проекты разные;
- кодовые базы разные;
- у каждой стороны свой локальный контекст;
- решение формируется совместно, а не через один общий репозиторий.

## Основные слои

### `domain/room`

Содержит доменную модель комнаты:

- статусы комнаты;
- типы сообщений;
- типы артефактов;
- decision log;
- финальные позиции;
- human feedback;
- правила переходов статусов.

Ключевой класс: `RoomService`.

Именно он отвечает за:

- открытие комнаты;
- подключение участников;
- объявление контекста;
- публикацию сообщений;
- сохранение артефактов и решений;
- расчёт статуса комнаты;
- завершение сессии.

### `infrastructure/persistence`

Слой хранения.

Теперь есть два backend-а:

- `JsonRoomRepository` как простой совместимый fallback;
- `SqliteRoomRepository` как основной backend для поиска и истории.

`SqliteRoomRepository`:

- хранит полный snapshot комнаты;
- строит отдельную search-проекцию;
- строит отдельную unified history-проекцию по сообщениям, артефактам, decision log и human feedback;
- использует встроенный `node:sqlite`, поэтому не требует внешнего нативного npm-драйвера.

`JsonRoomRepository` всё ещё:

- хранит полный снимок всех комнат в одном JSON;
- пишет через временный файл и атомарный rename;
- позволяет восстановить состояние после рестарта.

### `infrastructure/http`

HTTP слой состоит из:

- `TaskRoomHttpServer`
- `WebUiRenderer`

Он отвечает за:

- `health` endpoint;
- JSON API для UI;
- HTML-интерфейс комнаты;
- формы и action-кнопки для human message, artifacts, decision log и human feedback;
- быстрые действия координатора для типовых шагов без ручного заполнения форм;
- проксирование вызовов в доменный сервис;
- авторизацию через `Bearer token`.

### `infrastructure/a2a`

`A2A`-адаптер добавляет второй агентный транспорт поверх того же `RoomService`.

Он отвечает за:

- `Agent Card`;
- `JSON-RPC` endpoint;
- `HTTP+JSON/REST` endpoint;
- приём типизированных `task-room-command` сообщений;
- команды для `add_artifact` и `record_decision`;
- возврат структурированного результата в `data`-part ответа;
- `watch_room` как task/stream-подписку на обновления комнаты;
- webhook push-notifications для клиентов, которые не держат живой stream.

### `infrastructure/mcp`

MCP-адаптер не содержит отдельной бизнес-логики.

Он только маппит MCP tools на методы `RoomService`.

Это важно: у комнаты должен быть один источник правил, а не несколько несовместимых реализаций.

### `cli`

`TaskRoomCli` отвечает за пользовательский поток:

- запуск сервера;
- запуск `ngrok`;
- открытие комнаты;
- генерацию инструкций для коллеги;
- запись локального state;
- watcher и sync.

## Почему архитектура устроена так

### 1. Один доменный центр

Все решения о статусе комнаты живут в `RoomService`.

Это нужно, чтобы:

- CLI не считал статусы по-своему;
- UI не считал статусы по-своему;
- MCP не считал статусы по-своему.
- A2A не считал статусы по-своему.

### 2. UI, MCP и A2A — только адаптеры

HTTP UI, MCP и A2A являются транспортными слоями.

Они:

- принимают вход;
- вызывают доменный сервис;
- отдают результат обратно.

### 3. JSON persistence как первый шаг

Изначально для MVP был выбран простой слой хранения:

- легко локально поднимать;
- легко дебажить;
- легко переносить между машинами;
- не требует отдельной БД.

Следующим шагом он был расширен до `SQLite` backend-а с поиском и историей, не ломая доменную модель и transport-слои.

## Поток start

1. CLI поднимает локальный HTTP/MCP/A2A сервер.
2. CLI поднимает `ngrok`.
3. CLI открывает новую комнату через локальный MCP.
4. CLI запускает watcher.
5. CLI генерирует:
   - room URL
   - A2A URL-ы
   - join command
   - prompt для локального агента
   - prompt для peer-агента

## Поток общения

1. Агенты публикуют контекст через `declare_context`.
2. Агенты и люди публикуют ссылки, diff, логи и issue/PR в `artifacts`.
3. Агенты обмениваются типизированными сообщениями.
4. Люди могут писать общие сообщения в UI.
5. Агенты и люди фиксируют промежуточные договорённости в `decision log`.
6. Агенты публикуют финальные позиции.
7. Люди подтверждают или отклоняют решение.
